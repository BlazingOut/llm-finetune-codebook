在进入模型内部之前，我们需要统一“度量衡”。大模型的所有计算，本质上都是在对一种叫 **张量** 的数据进行**低精度**数学运算。

## 一、 张量 (Tensor)：运行在 GPU 上的多维数组

在 Python 中你用过列表（List），在科学计算中你可能用过 NumPy。
**张量 (Tensor)** 的工程定义很简单：它是一个**专门为 GPU 优化过的多维数组**。

- **0 维 (Scalar)**：一个孤立的数字，如 `loss = 0.5`。
- **1 维 (Vector)**：一行数字，如一个 Token 的 ID 序列。
- **2 维 (Matrix)**：一个表格，如词表（Vocab）或一张掩码表（Attention Mask）。
- **3 维及以上**：大模型最常见的形态。例如 `[Batch, Seq_Len, Hidden_Dim]`，代表“这一批有几句话，每句多少词，每个词多少维”。
$$\text{Scalar} \rightarrow \text{Vector} \rightarrow \text{Matrix} \rightarrow \text{Tensor}$$

**工程事实**：大模型微调的过程，就是不断地对这些张量进行乘法和加法。

## 二、 数值精度

在计算机里，存储一个数字是需要空间的。这个空间的大小，我们称为**精度 (Precision)**。
#### 1. FP32 (Full Precision / 单精度)

- **每个参数占用**：4 字节 (32 bit)。
- **现状**：精度极高，但太占显存。
- **物理映射**：一个 10 亿参数（1B）的模型，用 FP32 存储需要 **4GB** 显存。

#### 2. FP16 / BF16 (Half Precision / 半精度)

为了在有限的显存里塞进更大的模型，工程师们把 4 字节砍掉了一半，变成了 **2 字节 (16 bit)**。

| **精度类型** | **占用空间** | **特点**                             | **工程建议**              |
| -------- | -------- | ---------------------------------- | --------------------- |
| **FP16** | 2 字节     | 范围较窄，容易出现数值溢出（计算结果变 NaN）。          | 老旧显卡（如 V100）常用。       |
| **BF16** | 2 字节     | **大模型标配**。牺牲了微小的精度，但数值范围与 FP32 一致。 | **RTX 30/40 系列显卡首选**。 |

**物理映射公式**：

$$\text{模型基础显存} = \text{参数量} \times \text{每个参数的字节数}$$

- **结论**：使用 BF16 存储时，1B 参数仅需 **2GB** 显存。

## 三、 显存计算练习（避坑必备）

假设你要在 12GB 显存的 RTX 3060 上跑一个 **7B (70亿参数)** 的模型：

1. **加载模型 (BF16)**：$7 \times 2 = 14\text{GB}$。
2. **结论**：显存直接爆掉（OOM），你甚至连模型都打不开。

**这就是为什么我们要学习“量化（Quantization）”或选择更小的模型（如 Qwen-1.5B）的原因。**






## 四、Batch 与并行处理 —— 效率的秘密

### 1. 什么是 Batch（批次）？

**Batch** 的工程定义是：一次性送入 GPU 进行计算的样本数量。

- **形象理解**：如果你有 100 颗螺丝要拧，Batch=1 表示你一颗颗拧；Batch=10 表示你用一个带 10 个头的特制扳手一次拧 10 颗。    
- **张量形状的变化**：加入 Batch 后，数据的维度会从 `[Seq_Len, Hidden_Dim]` 变为 `[Batch_Size, Seq_Len, Hidden_Dim]`。

## 2. 为什么需要并行处理？

- **吞吐量**：GPU 拥有数千个核心，如果只计算一个 Token，大部分核心都在“摸鱼”。增加 Batch Size 可以让 GPU 满载，极大提升训练速度。
- **显存代价**：Batch Size 每翻一倍，计算时占用的**临时显存**也会近乎翻倍。
    - **工程金律**：微调时，Batch Size 是寻找“计算速度”与“显存上限”平衡点的核心杠杆。
